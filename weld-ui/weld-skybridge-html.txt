    <div class="hdr-row"><h1>weld.skybridge</h1><span class="ver">v1.2.0</span></div>
    <p class="lead">The bridge between a generator (sandboxed) and <strong>Weld Companion</strong> (a userscript running above the page). It lends a plugin two powers a sandbox cannot have on its own &mdash; storage that persists across every generator, and the user&rsquo;s own AI model &mdash; while never <em>requiring</em> the companion to be installed.</p>
    <p class="toc">on this page: <a href="#why">two contexts</a> &middot; <a href="#use">quick start</a> &middot; <a href="#api">api</a> &middot; <a href="#proto">protocol</a> &middot; <a href="#trust">trust</a> &middot; <a href="#fallback">fallback</a></p>
  </div>

  <h2 id="why">Two contexts, complementary powers</h2>
  <p>A Weld <strong>plugin</strong> runs inside the generator sandbox: universal (everyone who imports it gets it) but confined &mdash; no cross-generator persistence, no arbitrary network, no access to the page around it. <strong>Weld Companion</strong> runs as a userscript in the top frame: privileged (persistent storage, the user&rsquo;s AI key) but optional &mdash; most visitors won&rsquo;t have it.</p>
  <p>Skybridge joins them by <strong>composition with graceful degradation</strong>: a plugin stays fully self-sufficient and <em>opportunistically</em> gains the companion&rsquo;s powers when it detects one. Never the reverse &mdash; a plugin must not break without the companion.</p>

  <h2 id="use">Quick start</h2>
  <p>Import the plugin, then use it. Every privileged call returns a <em>result object</em> (or a value) and never throws, so you branch on data:</p>
  <pre><code><span class="c-c">// in your generator's $output body, after \{import:weld-skybridge-plugin\}</span>
<span class="c-k">var</span> sb = window.weld.skybridge;

sb.<span class="c-f">onConnect</span>(<span class="c-k">function</span> (info) \{            <span class="c-c">// fires if/when a companion links</span>
  console.<span class="c-f">log</span>(<span class="c-s">'linked, protocol'</span>, info.protocol);
\});

<span class="c-c">// storage works with OR without a companion (it falls back)</span>
<span class="c-k">await</span> sb.storage.<span class="c-f">set</span>(<span class="c-s">'slot1'</span>, state);
<span class="c-k">var</span> saved = <span class="c-k">await</span> sb.storage.<span class="c-f">get</span>(<span class="c-s">'slot1'</span>);

<span class="c-c">// AI needs a companion + the user's own model; check first</span>
<span class="c-k">if</span> (sb.<span class="c-f">has</span>(<span class="c-s">'ai'</span>)) \{
  <span class="c-k">var</span> r = <span class="c-k">await</span> sb.<span class="c-f">ai</span>(<span class="c-s">'Name a tavern.'</span>);
  <span class="c-k">if</span> (r.ok) <span class="c-f">use</span>(r.value);            <span class="c-c">// denied/absent -> r.ok is false</span>
\}</code></pre>

  <h4>Result reasons</h4>
  <p>When <code>ok</code> is <code>false</code>, <code>reason</code> is one of: <code>disconnected</code> (no companion), <code>unsupported</code> (companion lacks that capability), <code>denied</code> (user said no), <code>no-own-model</code> (AI: user hasn&rsquo;t configured a model), <code>timeout</code>, or <code>malformed</code>.</p>

  <h2 id="proto">Protocol versioning</h2>
  <p>The two halves ship independently and land in browsers at different times, so the handshake <strong>negotiates a protocol version</strong>. Each side advertises a <code>min</code>&ndash;<code>max</code> range; they settle on the highest both support. If there&rsquo;s no overlap, Skybridge stays <em>disconnected</em> rather than half-speaking a protocol it doesn&rsquo;t understand. This is separate from <code>version</code> (the plugin&rsquo;s own semver) &mdash; <code>protocol</code> is the wire contract.</p>
  <pre><code><span class="c-c">// the negotiation, conceptually</span>
plugin: hello  \{ protoMin:<span class="c-n">1</span>, protoMax:<span class="c-n">1</span> \}
anchor: here   \{ protoMin:<span class="c-n">1</span>, protoMax:<span class="c-n">2</span>, capabilities:\[<span class="c-s">'storage'</span>,<span class="c-s">'ai'</span>\] \}
<span class="c-c">// agreed protocol = min(pluginMax, anchorMax) within the shared range = 1</span></code></pre>

  <h2 id="trust">Trust model</h2>
  <p>The boundary isn&rsquo;t the <em>direction</em> of messages &mdash; <code>postMessage</code> is two-way, and anything on the page (including a hostile generator) can post at the top frame. The boundary is <strong>what the companion will act on</strong>. So:</p>
  <ul>
    <li><strong>Origin-checked both ways.</strong> Messages are only honoured from <code>perchance.org</code> / <code>*.perchance.org</code>.</li>
    <li><strong>Nonce-matched replies.</strong> Each request carries a nonce the reply must echo &mdash; no spoofing, no replay.</li>
    <li><strong>Scoped capabilities, not raw power.</strong> A plugin can ask for <em>&ldquo;an AI completion&rdquo;</em>, never <em>&ldquo;fetch this URL with my key&rdquo;</em>.</li>
    <li><strong>Per-capability consent.</strong> The first time a generator requests <code>storage</code> or <code>ai</code>, the companion asks the user &mdash; separately per capability &mdash; and remembers the choice for that generator.</li>
    <li><strong>Per-generator storage namespace.</strong> One generator cannot read another&rsquo;s stored keys.</li>
  </ul>
  <div class="warn"><strong>Secrets never cross the bridge.</strong> For <code>ai</code>, the prompt goes up and the companion runs the keyed call itself; only the completion comes back. The API key is never placed on the bridge in either direction.</div>

  <h2 id="fallback">Graceful fallback</h2>
  <p>With <strong>no companion installed</strong>, Skybridge stays disconnected and harmless:</p>
  <ul>
    <li><code>connected</code> is <code>false</code>, <code>has('ai')</code> is <code>false</code> &mdash; AI calls resolve <code>\{ ok:false, reason:'disconnected' \}</code>.</li>
    <li><code>storage</code> still works &mdash; it routes to <code>weld.persist</code> (IndexedDB, this browser) if present, else an in-memory map for the session.</li>
    <li><code>has('storage')</code> reports the truth, so a plugin <em>can</em> tell &ldquo;saved everywhere&rdquo; from &ldquo;saved just here&rdquo; &mdash; though most won&rsquo;t need to.</li>
  </ul>
  <div class="note">Design rule: a Skybridge-backed plugin must do something reasonable with zero companion, and something <em>better</em> with one. Build for the fallback first.</div>

  <h2>Changelog</h2>
  <h3>1.2.0</h3>
  <p>Storage gained a <strong>cross-device tier</strong>: when the host imports <code>weld-kv-plugin</code> (server-side key/value), skybridge uses it ahead of the local <code>weld.persist</code> fallback, so a companion-less generator can still persist across devices. The fallback order is now companion &rarr; kv &rarr; persist &rarr; memory. Internally, all four operations and <code>storage.backend()</code> now resolve through one <code>pickBackend()</code>, so the reported backend can never drift from where data is actually written.</p>
  <h3>1.1.0</h3>
  <p>Added the <strong>bus</strong> capability: <code>bus.publish(channel, message)</code> / <code>bus.subscribe(channel, fn)</code> for cross-tab, cross-generator pub/sub. The companion relays messages between subscribed generators in a tab and, over a <code>BroadcastChannel</code> on the shared <code>perchance.org</code> apex origin, across tabs &mdash; the cross-origin reach <code>weld.sync</code> can't provide. <code>has('bus')</code> reports availability; gated by per-generator consent like storage and AI. This is the transport <code>weld.swarm</code> rides on for multi-agent orchestration.</p>
  <h3>1.0.0</h3>
  <p>Initial release. Handshake with protocol negotiation (v1), origin- and nonce-checked request/response, capabilities <code>storage</code> and <code>ai</code>, per-capability consent (companion-side), per-generator storage namespace, graceful fallback to <code>weld.persist</code>/memory.</p>

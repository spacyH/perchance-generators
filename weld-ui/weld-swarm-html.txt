    <div class="hdr-row"><h1>weld.swarm</h1><span class="ver">v1.2.1</span></div>
    <p class="lead">Generator-side multi-agent orchestration &mdash; the DSL companion to the orchestrator that lives in <strong>Weld Companion</strong>. A generator joins a <strong>swarm</strong> with a role (<code>worker</code> = a tab running weld.crew, <code>user</code> = the driver), peers discover each other, address messages by id or role, and the driver fans tasks out across workers <strong>in parallel</strong> with a concurrency cap, awaiting results. The same routing logic the companion uses, available client-side.</p>
    <p class="lead" style="margin-bottom:0"><a href="https://perchance.org/weld-swarm-example">&rarr; Open SWARM // mission control (the live app)</a> (a self-contained local run, plus a two-tab mode over weld.sync).</p>
    <p class="toc">on this page: <a href="#why">concept</a> &middot; <a href="#use">quick start</a> &middot; <a href="#transport">transport</a> &middot; <a href="#api">api</a> &middot; <a href="#status">status</a></p>
  </div>

  <h2 id="why">Concept</h2>
  <p>weld.crew makes <em>one</em> generator agentic. weld.swarm makes <em>several</em> cooperate: a <strong>user</strong> peer (a chat a human drives) hands a goal to the crew, which splits it and dispatches subtasks to <strong>worker</strong> peers &mdash; each a tab running weld.crew &mdash; running them in parallel and collecting the results. The tab that calls <code>dispatch()</code> acts as the orchestrator for that exchange; the heavy lifting (a privileged cross-tab bus) is what the companion adds.</p>

  <h2 id="use">Quick start</h2>
  <pre><code><span class="c-c">// worker generator (also imports weld.crew):</span>
<span class="c-k">var</span> a = window.weld.crew.<span class="c-f">create</span>(\{ workspace: <span class="c-s">'crew'</span> \});
window.weld.swarm.<span class="c-f">create</span>(\{ role: <span class="c-s">'worker'</span> \}).<span class="c-f">serve</span>(<span class="c-k">function</span> (task) \{
  <span class="c-k">return</span> a.<span class="c-f">run</span>(task.goal);            <span class="c-c">// answer each task by running the agent</span>
\});

<span class="c-c">// user / driver generator (the chat):</span>
<span class="c-k">var</span> crew = window.weld.swarm.<span class="c-f">create</span>(\{ role: <span class="c-s">'user'</span> \});
<span class="c-k">var</span> results = <span class="c-k">await</span> crew.<span class="c-f">dispatchMany</span>(subtasks, \{ concurrency: <span class="c-n">3</span> \});</code></pre>

  <h2 id="transport">Transport (auto-selected)</h2>
  <p>The crew picks the best channel available, in order: the <strong>skybridge bus</strong> when the companion provides it (cross-generator, cross-tab — now available); else <strong>weld.sync</strong> (same-origin tabs of one generator); else in-memory (single tab). Same generator code either way &mdash; it upgrades from same-origin to cross-generator the moment the companion bus is present.</p>
  <div class="warn"><strong>Today vs. full vision.</strong> Over weld.sync alone, a &ldquo;crew&rdquo; means multiple tabs of <em>one</em> generator (weld.sync is same-origin). True cross-generator crews need the companion&rsquo;s bus capability. Also: trigger the plugin once with <code>root.weldSwarm()</code> &mdash; imported <code>$output</code> is not auto-run.</div>

  <h2 id="api">API</h2>
  <span class="sig">window.weld.swarm.create(opts) -> crew</span>
  <p><code>opts</code>: <code>role</code> (<code>'user'</code> | <code>'worker'</code>), <code>channel</code> (bus/sync channel name), <code>concurrency</code>, <code>peerTTL</code>, <code>heartbeatMs</code>, <code>transport</code> (inject a custom one), <code>id</code>.</p>
  <table>
    <tr><th>Member</th><th>What it does</th></tr>
    <tr><td><code>dispatch(workerId, task, timeoutMs?)</code></td><td>Assign one task to one worker; resolves its result (or rejects on timeout).</td></tr>
    <tr><td><code>dispatchMany(tasks, \{concurrency, timeoutMs\})</code></td><td>Fan tasks across live workers with a concurrency cap; resolves results in order.</td></tr>
    <tr><td><code>serve(handler)</code></td><td>Worker convenience: answer each <code>task</code> by running <code>handler(payload)</code> (typically a weld.crew run).</td></tr>
    <tr><td><code>on(type, fn) / send(to, type, payload)</code></td><td>Low-level pub/sub. <code>to</code> = a peer id, a role, or <code>'*'</code>.</td></tr>
    <tr><td><code>peers() / workers() / users()</code></td><td>The live registry, filtered by role (presence-pruned).</td></tr>
    <tr><td><code>leave() / transport / id / role</code></td><td>Announce departure / which transport is active / this peer&rsquo;s identity.</td></tr>
  </table>

  <h2 id="status">Status</h2>
  <p>Stage 1 of the orchestrator. The routing core (registry, addressing, parallel dispatch, concurrency cap, presence, timeouts) is built and tested. A live demo is at <a href="https://perchance.org/weld-swarm-example">weld-swarm-example</a> &mdash; a local run (driver + workers in one tab) and a two-tab Worker/User mode over weld.sync. Cross-generator operation rides the companion's <code>bus</code> capability (now shipped): when Weld Companion is installed, weld.swarm coordinates workers across <em>different</em> generators and tabs over the apex <code>BroadcastChannel</code>; with no companion it falls back to weld.sync (same-origin tabs).</p>

  <h2>Changelog</h2>
  <h3>1.2.1</h3>
  <p><code>serve()</code> is now idempotent (replaces its task handler instead of stacking a new one each call) and dedupes duplicate task deliveries by correlation id. Fixes workers running &mdash; and replying/streaming &mdash; a task twice when serve was re-invoked or the bus delivered a message more than once.</p>
  <h3>1.2.0</h3>
  <p>Resilient dispatch: <code>dispatchMany</code> now re-queries the live worker set each cycle and reassigns a timed-out or unreachable subtask to a different worker (up to <code>attempts</code>, default 3) &mdash; so a backgrounded, frozen tab no longer makes its subtask fail or vanish. Pairs with the app's keep-awake (silent audio) for Node tabs.</p>
  <h3>1.1.0</h3>
  <p>Cross-generator operation, for real: weld.swarm now rides the companion's <code>bus</code> capability when present, coordinating workers across <em>different</em> generators and tabs (previously limited to same-origin tabs over weld.sync). The transport guard requires a connected companion advertising <code>bus</code>; with none it falls back to weld.sync, then in-memory, so a single-tab demo still runs.</p>
  <h3>1.0.0</h3>
  <p>Initial release. Role-based registry, message routing by id/role/broadcast, parallel <code>dispatchMany</code> with concurrency cap, <code>serve</code> for workers, presence heartbeat/prune, dispatch timeouts. Transport auto-selects skybridge bus &rarr; weld.sync &rarr; memory.</p>

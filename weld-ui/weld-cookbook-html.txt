</nav><div class="cb-main"><h2 id="what">What Weld is</h2>
<p>Weld is a suite of small, composable plugins for building AI generators on Perchance. Each plugin is itself a tiny Perchance generator that exposes a clean JavaScript API on a shared global, <code>window.weld</code>. Your app imports the ones it needs and calls them; they don't fight each other, and they degrade gracefully when an optional dependency is missing.</p>
<p>This cookbook is the part that isn't on the tin: how a Perchance plugin is actually structured, how importing and exporting really work, and the platform quirks that cost every author hours the first time. If you only read one thing, read <a href="#braces">Brace interception</a>. For how the platform itself is shaped &mdash; domains, the root bridge, the sandbox, and the live Platform Monitor &mdash; see the companion <a href="https://perchance.org/weld-platform">Perchance Platform</a> reference.</p>
<h2 id="anatomy">Anatomy of a plugin</h2>
<p>A Perchance generator has two editable areas: a <strong>code editor</strong> (the Perchance DSL, which can contain JavaScript) and an <strong>HTML panel</strong>. A Weld plugin lives almost entirely in the code editor. It has two parts: a <code>$meta</code> block (title, description, tags) and a <code>$output()</code> function whose body runs once and installs the plugin's API.</p><pre class="cb-code">$meta
  title       = weld.example
  description = One sentence on what it does.
  tags        = weld, example, plugin

$output() =&gt;
  'use strict';

  // 1. ensure the shared namespace exists
  if (!window.weld) window.weld = &#123;&#125;;

  // 2. idempotent guard: if already installed, hand back the same instance
  if (window.weld.example) return window.weld.example;

  // 3. private state lives in this closure
  var count = 0;
  function bump() &#123; count += 1; return count; &#125;

  // 4. install the API on the namespace AND return it
  window.weld.example = &#123;
    bump: bump,
    value: function () &#123; return count; &#125;,
    version: '1.0.0'
  &#125;;
  return window.weld.example;</pre><p>Five things make this a well-behaved plugin:</p>
<ul>
<li><strong><code>'use strict';</code></strong> first &mdash; catches silent mistakes.</li>
<li><strong>The namespace check</strong> &mdash; never assume <code>window.weld</code> exists; another plugin may load first or not at all.</li>
<li><strong>The idempotent guard</strong> &mdash; a generator can evaluate <code>$output</code> more than once. Returning the existing instance keeps state intact and avoids re-running setup.</li>
<li><strong>A private closure</strong> &mdash; everything internal stays local; only the returned object is public.</li>
<li><strong>A semver <code>version</code></strong> &mdash; so consumers can feature-detect.</li>
</ul>
<h2 id="make">Make your first plugin</h2>
<ol class="cb-steps">
<li>On Perchance, create a new generator. Name it with your plugin's slug, e.g. <code>weld-example-plugin</code> &mdash; the slug is how others will import it.</li>
<li>Paste the skeleton above into the <strong>code editor</strong> (not the HTML panel). The <code>$output()</code> body is where your logic goes.</li>
<li>Save. Your plugin is now importable by anyone as <code>&#123;import:weld-example-plugin&#125;</code>.</li>
<li>To try it, open the HTML panel and add a tiny script that waits for the API and calls it:</li>
</ol><pre class="cb-code">&lt;script&gt;
  function boot()&#123;
    if (!window.weld &#124;&#124; !window.weld.example) return setTimeout(boot, 100);
    console.log('bumped to', window.weld.example.bump());
  &#125;
  boot();
&lt;/script&gt;</pre><div class="cb-note">Plugins install asynchronously relative to your HTML, so always poll for the API (<code>setTimeout(boot, 100)</code>) rather than assuming it's ready on first paint. This is the single most common "my plugin is undefined" bug.</div>
<h2 id="import">Imports &amp; exports &mdash; how they really work</h2>
<h3>Importing</h3>
<p>You pull a plugin in with an <strong>import directive</strong>, assigned to a name, at the <strong>top level</strong> of the code editor &mdash; above <code>$output()</code>:</p><pre class="cb-code">aiTextPlugin = &#123;import:ai-text-plugin&#125;
weldStream   = &#123;import:weld-stream-plugin&#125;
weldClean    = &#123;import:weld-clean-plugin&#125;

$output() =&gt;
  ...</pre><p>Two things matter here, and both bite newcomers:</p>
<ul>
<li>An import directive is <strong>only valid as a top-level assignment</strong>. Writing one inside the <code>$output()</code> body, inside a string, or inside a comment does <em>not</em> import anything &mdash; instead the parser intercepts it (see <a href="#braces">Brace interception</a>). Imports go at the top, full stop.</li>
<li>Importing a plugin <strong>runs its <code>$output()</code></strong>, which installs its API on <code>window.weld</code>. The name you assign (<code>weldStream</code>) is mostly a formality for the platform; in practice you reach the plugin through the global, e.g. <code>window.weld.stream</code>, not through that variable.</li>
</ul>
<h3>Exporting</h3>
<p>There is no special "export" keyword. A plugin "exports" simply by assigning its API object onto <code>window.weld.&lt;name&gt;</code> (and returning it). That global <em>is</em> the public surface. Any other plugin or your app's HTML can then read <code>window.weld.&lt;name&gt;</code> once it has loaded.</p>
<div class="cb-note good">Mental model: <strong>import = "run this plugin so its global appears"</strong>; <strong>export = "put my API on the shared global."</strong> The named import variable is not the API &mdash; <code>window.weld</code> is.</div>
<h3>Why a single shared namespace</h3>
<p>Because every plugin attaches to one <code>window.weld</code> object, plugins can find each other at runtime without importing each other in code. <code>weld.memory</code> can use <code>weld.embed</code> if it happens to be present, and quietly fall back if not. That's the whole composition story &mdash; and it's why the idempotent guard matters: load order is not guaranteed.</p>
<h2 id="deps">Dependency injection between plugins</h2>
<p>If plugin A needs plugin B, it does <strong>not</strong> import B in its body (you can't &mdash; imports are top-level only, and a plugin shouldn't force its consumers to load B). Instead, A exposes an <code>attach()</code> method, and the app wires them together at boot:</p><pre class="cb-code">// inside weld.memory's body:
var deps = &#123; embed: null &#125;;
function attach(d) &#123; if (d &amp;&amp; d.embed) deps.embed = d.embed; &#125;
function getEmbed() &#123; return deps.embed &#124;&#124; (window.weld &amp;&amp; window.weld.embed) &#124;&#124; null; &#125;

// ...exposed as part of the API: &#123; attach: attach, ... &#125;

// in your app's boot, after both plugins have loaded:
weld.memory.attach(&#123; embed: weld.embed &#125;);</pre><p>This keeps dependencies <strong>soft</strong>: <code>weld.memory</code> works on its own (keyword recall) and gets better when <code>weld.embed</code> is attached (semantic recall). The <code>getEmbed()</code> helper even falls back to reading the global directly, so attaching is optional. Design every cross-plugin dependency this way and your plugin never hard-crashes because something wasn't loaded.</p><div class="cb-note">Rule of thumb: <strong>required</strong> building blocks (like the AI plugin a generator wraps) are imported at the top; <strong>optional</strong> enhancements are injected via <code>attach()</code> and feature-detected.</div>
<h2 id="braces">Brace interception &mdash; the #1 footgun</h2>
<p>Before your code runs, Perchance scans the <strong>raw source</strong> &mdash; including the insides of <code>&lt;script&gt;</code> blocks, string literals, <em>and</em> comments &mdash; and rewrites anything that looks like its own template syntax. A token sitting harmlessly in a JavaScript string can be silently mangled, producing a bug with no stack trace.</p>
<p>These tokens are intercepted everywhere:</p>
<table class="cb-table">
<tr><th>Token</th><th>What the parser thinks</th><th>Safe alternative</th></tr>
<tr><td><code>&#91;&#91;</code></td><td>nested list / shortcode</td><td>build at runtime, or add a space: <code>&#91; &#91;</code></td></tr>
<tr><td><code>&#123;&#123;</code></td><td>advanced template syntax</td><td>space them: <code>&#123; &#123;</code> (doubled <em>closing</em> is fine)</td></tr>
<tr><td><code>&#123;import:x&#125;</code> (not top-level)</td><td>an import directive</td><td>keep imports at the top; escape if quoting one</td></tr>
<tr><td><code>&#123;a&#124;b&#124;c&#125;</code> (tight, no spaces)</td><td>pick one at random</td><td>add spaces, or escape with entities</td></tr>
<tr><td><code>&#123;3&#125;</code> / <code>&#123;2,5&#125;</code></td><td>a repeat/count token</td><td>fine inside a real regex literal; else escape</td></tr>
<tr><td><code>&#123;word&#125;</code> (bare)</td><td>a variable / list reference</td><td>safe in real JS; escape in prose/strings</td></tr>
</table>
<p>And these look similar but are <strong>safe</strong> &mdash; don't waste time avoiding them:</p>
<ul>
<li>Object literals <em>with spaces</em>: <code>&#123; name: 1 &#125;</code></li>
<li>Logical-or in braces with spaces: <code>&#123; a &#124;&#124; b &#125;</code></li>
<li>Regex character classes: <code>&#91;A-Z&#93;</code></li>
<li>Single-identifier array indexing: <code>arr&#91;i&#93;</code></li>
<li>A doubled <em>closing</em> brace from nested blocks: <code>&#125;&#125;</code></li>
</ul>
<h3>Three ways out</h3>
<p>When you genuinely need a trap token, pick one:</p>
<ol>
<li><strong>Escape it</strong> (for prose/HTML you want shown literally) &mdash; replace the chars with HTML entities, e.g. <code>&amp;#123;a&amp;#124;b&amp;#125;</code> renders as the literal text.</li>
<li><strong>Build it at runtime</strong> (for strings your code needs) &mdash; assemble from char codes so the literal never appears in source: <code>String.fromCharCode(123) + 'x' + String.fromCharCode(125)</code>.</li>
<li><strong>Space it</strong> (for JS syntax) &mdash; a space inside the braces is enough for the parser to read it as code, not a template.</li>
</ol>
<div class="cb-note warn">The plugin <code>weld.safe</code> automates all of this: <code>scan(code)</code> reports every trap with a fix, <code>escape(str)</code> does option 1, and <code>build(token)</code> / <code>builderSnippet(token)</code> do option 2. If you write plugins regularly, keep it open.</div>
<h2 id="ai">AI plugin gotchas</h2>
<p>The <code>ai-text-plugin</code> is what most generators wrap. Its sharp edges:</p>
<table class="cb-table">
<tr><th>Gotcha</th><th>What to do</th></tr>
<tr><td>Results are a <strong>boxed String</strong> object, not a primitive</td><td>Never compare with <code>===</code>. Coerce: <code>String(result)</code> or read <code>result.generatedText</code>.</td></tr>
<tr><td><code>stopReason</code> is one of <code>natural</code> / <code>artificial</code> / <code>error</code> / <code>user</code></td><td>Branch on it; "artificial" means a stop-sequence/limit hit, so you may want to continue or trim.</td></tr>
<tr><td>The broker is <strong>serial</strong> (concurrency 1)</td><td>Calls queue; don't fire dozens in parallel expecting speed. Cache repeats (see <code>weld.cache</code>).</td></tr>
<tr><td>Context budget &asymp; <strong>6000 tokens</strong>, reserve ~800 for the reply</td><td>Pack context to a budget (see <code>weld.tokens</code> / <code>weld.prompt</code>), keep the prompt prefix stable for cache hits.</td></tr>
<tr><td>At most <strong>20 stop sequences</strong>; output ceiling ~900 tokens</td><td>Plan long generations as multiple calls.</td></tr>
</table>
<div class="cb-note">Most of these are smoothed over by <code>weld.stream</code> (lifecycle + normalization), <code>weld.tokens</code> (budgeting), and <code>weld.clean</code> (trimming a truncated tail). The boxed-String trap in particular has bitten nearly every Perchance AI app at least once.</div>
<h2 id="t2i">Image plugin gotchas</h2>
<p>For <code>text-to-image-plugin</code>:</p>
<ul>
<li>The awaited result is a <strong>boxed String</strong> carrying <code>canvas</code>, <code>dataUrl</code>, and <code>inputs</code> &mdash; same coercion caution as text.</li>
<li>Only four resolutions are supported: <code>512&times;512</code>, <code>512&times;768</code>, <code>768&times;512</code>, <code>768&times;768</code>. Anything else is rejected.</li>
<li>An <strong>empty prompt hangs forever</strong> &mdash; guard against it before calling.</li>
<li><code>removeBackground:true</code> runs RMBG-1.4 client-side, but <strong>only post-processes images it generated</strong>, not images you uploaded. To cut out an upload, use <code>weld.background</code>.</li>
</ul>
<div class="cb-note"><code>weld.image</code> wraps resolution validation and result normalization; <code>weld.background</code> does client-side cutout/replace for any image (including uploads and batches); <code>weld.promptforge</code> evolves the prompts you feed in.</div>
<h2 id="convention">The 4-file convention</h2>
<p>Every Weld plugin ships as four generators, by slug:</p>
<table class="cb-table">
<tr><th>File</th><th>Goes in</th><th>Purpose</th></tr>
<tr><td><code>weld-&lt;slug&gt;-plugin</code></td><td>code editor</td><td>the plugin itself (<code>$meta</code> + <code>$output</code>)</td></tr>
<tr><td><code>weld-&lt;slug&gt;-plugin</code> HTML panel</td><td>HTML panel</td><td>the docs page</td></tr>
<tr><td><code>weld-&lt;slug&gt;-example</code></td><td>both panels</td><td>an interactive demo</td></tr>
</table>
<p>Import as <code>&#123;import:weld-&lt;slug&gt;-plugin&#125;</code>; docs live at <code>perchance.org/weld-&lt;slug&gt;-plugin</code> and the demo at <code>perchance.org/weld-&lt;slug&gt;-example</code>. Keeping this consistent is what lets the hub index and manifest stay in sync automatically.</p>
<h2 id="recipes">Recipes</h2>
<h3>A streaming chat bot</h3>
<p>Import the AI plugin plus stream, tokens, clean, and memory. Stream the reply, budget the history, clean the output, and let memory recall facts:</p><pre class="cb-code">aiTextPlugin = &#123;import:ai-text-plugin&#125;
weldStream   = &#123;import:weld-stream-plugin&#125;
weldTokens   = &#123;import:weld-tokens-plugin&#125;
weldClean    = &#123;import:weld-clean-plugin&#125;
weldMemory   = &#123;import:weld-memory-plugin&#125;

// in your send handler (HTML side):
var context = weld.memory.inject(userText);          // relevant facts
var kept    = weld.tokens.clamp(history, context);   // fit the budget
weld.stream.run(&#123;
  instruction: context + '\n' + render(kept),
  onChunk: function (delta, full) &#123; paint(full); &#125;
&#125;).then(function (res) &#123;
  var reply = weld.clean.clean(res, &#123; playerName: you, roles: true &#125;);
  show(reply);
  weld.memory.observe(reply);                         // learn new facts
&#125;);</pre><h3>An image generator with evolving prompts</h3><pre class="cb-code">aiTextPlugin     = &#123;import:ai-text-plugin&#125;
textToImagePlugin = &#123;import:text-to-image-plugin&#125;

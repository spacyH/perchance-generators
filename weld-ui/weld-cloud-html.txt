    <div class="hdr-row"><h1>weld.cloud</h1><span class="ver">v0.1.5 &middot; preview</span></div>
    <p class="lead">Genuinely cross-device key&ndash;value storage for Perchance. It rides <strong>Weld Companion</strong>&rsquo;s <code>fetch</code> capability to a backend <em>you</em> configure, so saves can follow a user across devices &mdash; and it reports its real scope honestly, never claiming sync it cannot deliver.</p>
    <p class="toc">on this page: <a href="#why">why</a> &middot; <a href="#use">quick start</a> &middot; <a href="#api">api</a> &middot; <a href="#backend">backend</a> &middot; <a href="#scope">scope &amp; honesty</a> &middot; <a href="#status">status</a> &middot; <a href="#log">changelog</a></p>
  </div>

  <h2 id="why">Why this exists</h2>
  <p>Perchance&rsquo;s own cross-device options aren&rsquo;t usable as a durable store from a normal generator today. <code>kv-plugin</code> (what <code>weld.kv</code> wraps) is <strong>local IndexedDB</strong>, partitioned per generator &mdash; one device. <code>server-plugin</code> is a relay whose embed endpoint is currently returning HTTP&nbsp;526, so it won&rsquo;t even connect. Skybridge&rsquo;s <code>storage</code> capability is shared across generators but is still <em>one device</em> (the companion&rsquo;s storage is local to that browser).</p>
  <p>The one primitive that genuinely reaches another device is the companion&rsquo;s <strong><code>fetch</code></strong> capability: the userscript runs in the top frame and can make arbitrary cross-origin requests. weld.cloud rides it to a backend <em>you</em> own &mdash; your server, or a tiny Cloudflare Worker / Deno&nbsp;Deploy KV. Cross-device works exactly when (a)&nbsp;the companion is installed on each device and (b)&nbsp;each device is configured with the same backend &mdash; and <code>mode()</code> tells you the truth either way.</p>

  <h2 id="use">Quick start</h2>
  <p>Import the plugin (it needs <code>weld-skybridge-plugin</code> for the bridge), configure your backend once at boot, then use foldered get/set:</p>
  <pre><code><span class="c-c">// lists panel: cloud = \{import:weld-cloud-plugin\}, plus \{import:weld-skybridge-plugin\}</span>
<span class="c-k">var</span> cloud = window.weld.cloud;

<span class="c-c">// once at boot &mdash; point at YOUR backend (public HTTPS, header auth)</span>
cloud.<span class="c-f">configure</span>(\{
  baseUrl: <span class="c-s">'https://my-worker.example.workers.dev'</span>,
  headers: \{ Authorization: <span class="c-s">'Bearer '</span> + MY_TOKEN \}
\});

<span class="c-k">var</span> saves = cloud.<span class="c-f">folder</span>(<span class="c-s">'save-data'</span>);
<span class="c-k">await</span> saves.<span class="c-f">set</span>(<span class="c-s">'slot1'</span>, \{ level: <span class="c-n">4</span>, gold: <span class="c-n">120</span> \});   <span class="c-c">// durable + cross-device when mode()==='cloud'</span>
<span class="c-k">var</span> s = <span class="c-k">await</span> saves.<span class="c-f">get</span>(<span class="c-s">'slot1'</span>, \{\});

<span class="c-c">// be honest with users about scope</span>
<span class="c-k">if</span> (cloud.<span class="c-f">isCrossDevice</span>()) <span class="c-f">showSyncBadge</span>();</code></pre>

  <h2 id="backend">The backend contract</h2>
  <p>weld.cloud&rsquo;s default <code>rest</code> adapter maps each operation to a small REST shape that <em>you</em> implement (auth via your configured headers; values are wrapped as <code>\{"v": ...\}</code> so any JSON type round-trips):</p>
  <pre><code>PUT    \{baseUrl\}/\{ns\}/\{folder\}/\{key\}    body: \{"v": ...\}    <span class="c-c">-&gt; 2xx</span>
GET    \{baseUrl\}/\{ns\}/\{folder\}/\{key\}                       <span class="c-c">-&gt; \{"v": ...\}  (404 = absent)</span>
DELETE \{baseUrl\}/\{ns\}/\{folder\}/\{key\}                       <span class="c-c">-&gt; 2xx</span>
GET    \{baseUrl\}/\{ns\}/\{folder\}                             <span class="c-c">-&gt; \["key1","key2", ...\]</span></code></pre>
  <div class="warn"><strong>Three constraints from the fetch capability:</strong> the backend must be <strong>public HTTPS</strong> (private/loopback hosts are blocked); <strong>auth goes in a header</strong>, never a cookie (requests are anonymous); and responses are <strong>capped at ~200KB</strong> &mdash; an oversize read is treated as unreliable and served from the local mirror.</div>
  <p>A ~20-line Cloudflare Worker over KV satisfies this contract; see the design notes for a copy-paste reference and a JSONBin/Supabase no-server path.</p>

  <h2 id="scope">Scope &amp; honesty</h2>
  <p>This plugin exists because the previous cross-device story in the suite was wrong: <code>weld.kv</code> once advertised &ldquo;follows the user across devices&rdquo; while actually writing to local IndexedDB. weld.cloud will not repeat that. It carries no implicit promise &mdash; the only way to get cross-device behaviour is to configure a real shared backend, and the plugin reports exactly what it&rsquo;s doing:</p>
  <ul>
    <li><code>mode()==='cloud'</code> &mdash; a backend is configured and the companion can fetch; writes are durable and shared across devices that share the config.</li>
    <li><code>mode()==='memory'</code> &mdash; no backend (or no companion); the API still works against a per-session in-memory mirror, but nothing leaves the tab.</li>
  </ul>
  <div class="note">Design rule: behave reasonably with no backend, and <em>better</em> with one &mdash; and always tell the truth about which you&rsquo;re in.</div>

  <h2 id="status">Status</h2>
  <p>weld.cloud is a <strong>v0.1 preview</strong>. The local path (folder API, in-memory mirror, honest mode reporting, ~200KB truncation fallback) is verified end-to-end. The cross-device path is implemented and depends only on a configured backend. Slated for <strong>v0.2</strong>: a durable local tier (via <code>weld.persist</code> or the skybridge <code>storage</code> capability, once verified), value chunking above 200KB, and a corrected <code>delete()</code> return in memory mode.</p>

  <h2 id="log">Changelog</h2>
  <h3>0.1.5</h3>
  <p>Mirrors are cached per folder name, so every <code>folder('x')</code> call shares one store &mdash; fixes a round-trip bug where <code>set</code> and <code>get</code> hit different in-memory maps.</p>
  <h3>0.1.3</h3>
  <p>Config now persists via <code>localStorage</code>, and the mirror is self-contained. Removed the dependency on <code>weld.kv</code>, which could not reach <code>kv-plugin</code> in practice (<code>root.kv</code> surfaced as the unevaluated list), so it threw on every op.</p>
  <h3>0.1.0</h3>
  <p>Initial preview: cross-device key&ndash;value over the skybridge <code>fetch</code> capability to a user-configured backend; <code>folder</code> API with <code>set</code>/<code>get</code>/<code>has</code>/<code>keys</code>/<code>delete</code>/<code>getMany</code>; write-through local mirror; honest <code>mode()</code> / <code>isCrossDevice()</code>.</p>

    <div class="hdr-row"><h1>weld.crew</h1><span class="ver">v1.1.0</span></div>
    <p class="lead">A tool-using AI agent that runs <em>inside</em> a Perchance generator, on the user&rsquo;s own model, over <a href="https://perchance.org/weld-skybridge-plugin">weld.skybridge</a>. It runs a ReAct loop &mdash; reason, call a tool, read the result, repeat &mdash; and keeps its state in the companion&rsquo;s storage: a long-term <strong>memory</strong> it can write and search, a per-task <strong>scratchpad</strong> of thoughts, and named <strong>workspaces</strong> that survive reloads, so a run can be paused, resumed, or inspected later.</p>
    <p class="toc">on this page: <a href="#why">concept</a> &middot; <a href="#use">quick start</a> &middot; <a href="#contract">output contract</a> &middot; <a href="#api">api</a> &middot; <a href="#tools">tools</a> &middot; <a href="#state">memory &amp; workspaces</a> &middot; <a href="#debug">live debug</a> &middot; <a href="#fallback">fallback</a></p>
  </div>

  <h2 id="why">Concept</h2>
  <p>A generator can already call an AI once. An <strong>agent</strong> calls it in a loop, deciding what to do next from what it has seen so far &mdash; saving facts, calling tools, building toward a goal. weld.crew supplies that loop. Reasoning runs on the user&rsquo;s own model through skybridge (the key never crosses), and every bit of state lives in skybridge storage, so the agent&rsquo;s memory and half-finished work persist across reloads and across runs &mdash; something a sandboxed generator cannot do alone.</p>
  <div class="warn"><strong>Trigger it once.</strong> An imported plugin&rsquo;s <code>$output</code> is a function <em>definition</em>; Perchance does not auto-run it. Call <code>root.weldCrew()</code> (and <code>root.weldSkybridge()</code>) once in your panel JS or <code>window.weld.crew</code> stays <code>undefined</code>.</div>

  <h2 id="use">Quick start</h2>
  <pre><code><span class="c-c">// generator lists: \{import:weld-skybridge-plugin\} and \{import:weld-crew-plugin\}</span>
<span class="c-c">// panel JS:</span>
<span class="c-k">if</span> (<span class="c-k">typeof</span> root.weldSkybridge === <span class="c-s">'function'</span>) root.<span class="c-f">weldSkybridge</span>();
<span class="c-k">if</span> (<span class="c-k">typeof</span> root.weldCrew === <span class="c-s">'function'</span>) root.<span class="c-f">weldCrew</span>();

<span class="c-k">var</span> a = window.weld.crew.<span class="c-f">create</span>(\{ workspace: <span class="c-s">'quest'</span> \});

a.<span class="c-f">registerTool</span>(\{ name: <span class="c-s">'roll'</span>, description: <span class="c-s">'roll dice like 2d6'</span>,
  args: \{ dice: <span class="c-s">'string'</span> \}, run: <span class="c-k">function</span> (x) \{ <span class="c-k">return</span> <span class="c-f">roll</span>(x.dice); \} \});

<span class="c-k">var</span> out = <span class="c-k">await</span> a.<span class="c-f">run</span>(<span class="c-s">'Plan a tavern encounter, roll for initiative.'</span>, \{
  onStep: <span class="c-k">function</span> (s) \{ <span class="c-f">render</span>(s); \}      <span class="c-c">// start/thought/action/observation/final</span>
\});
<span class="c-c">// out.answer, out.runId  -> resume later with a.resume(out.runId)</span></code></pre>

  <h2 id="contract">Output contract</h2>
  <p>Each step, the loop asks the model for <strong>one JSON object</strong>. To call a tool: <code>\{ "thought": "&hellip;", "tool": "&lt;name&gt;", "args": \{ &hellip; \} \}</code>. To finish: <code>\{ "thought": "&hellip;", "final": "&lt;answer&gt;" \}</code>. The parser tolerates prose around the JSON and <code>```json</code> fences, and if the model just answers in prose it is treated as the final answer. Unknown tool names come back as an observation error so the model can recover.</p>

  <h2 id="api">API</h2>
  <span class="sig">window.weld.crew.create(opts) -> agent</span>
  <p><code>opts</code>: <code>workspace</code> (state namespace, default <code>'default'</code>), <code>maxSteps</code> (default <code>8</code>), <code>system</code> (extra instructions), <code>model</code> (passed to <code>sb.ai</code>), <code>tools</code> (array of specs), <code>reason</code> (custom reasoner; overrides the bridge model).</p>
  <table>
    <tr><th>Member</th><th>What it does</th></tr>
    <tr><td><code>run(goal, \{onStep, maxSteps, runId\})</code></td><td>Run the loop. Resolves <code>\{ ok, answer, runId, steps \}</code>. <code>onStep</code> fires per event.</td></tr>
    <tr><td><code>resume(runId, \{maxSteps\})</code></td><td>Continue a saved run where it left off (give a larger budget than it stopped at).</td></tr>
    <tr><td><code>registerTool(\{name, description, args, run\})</code></td><td>Add a tool the agent may call; <code>run(args)</code> may be async and returns the observation.</td></tr>
    <tr><td><code>memory.write(text, tags?) / search(q, tags?) / list()</code></td><td>Durable long-term memory. <code>write</code> returns an id; <code>search</code> ranks by relevance (whole-phrase, then per-term overlap, then tags).</td></tr>
    <tr><td><code>scratchpad.read() / write(note) / clear()</code></td><td>The agent&rsquo;s per-run thoughts, persisted every step.</td></tr>
    <tr><td><code>runs() / getRun(id)</code></td><td>List run records / fetch one (goal, steps, status, result).</td></tr>
    <tr><td><code>tools() / backend() / stop()</code></td><td>List tools / where state is going (<code>companion</code>/<code>persist</code>/<code>memory</code>) / abort a run.</td></tr>
  </table>

  <h2 id="tools">Tools</h2>
  <p>Built-in (no companion capability beyond storage): <code>think</code>, <code>memory.write</code>, <code>memory.search</code> (ranked by relevance), <code>memory.list</code>, <code>time.now</code>. Two web tools: <code>web.fetch</code> (uses the companion <code>fetch</code> capability if present, else <code>weld.fetch</code>; it extracts readable text from HTML, and given a <code>query</code> it keeps the query-relevant passages instead of blindly truncating the head) and <code>web.search</code> (a short result list of <code>\{title, url, snippet\}</code> via the companion <code>search</code> capability). Register your own with <code>registerTool</code> &mdash; that is how the agent orchestrates the generator&rsquo;s own powers (dice, lorebook lookup, image generation, sheet edits).</p>

  <h2 id="state">Memory &amp; workspaces</h2>
  <p>State lives in skybridge storage under <code>agent:&lt;workspace&gt;:&lt;ns&gt;</code> &mdash; <code>:mem</code> (long-term memory), <code>:scratch</code> (this run&rsquo;s thoughts), <code>:run:&lt;id&gt;</code> (a run record), <code>:runs</code> (the index). Because the run record is written every step, a run survives a reload and <code>resume(runId)</code> picks it up. Switch <code>workspace</code> to keep separate bodies of work apart; with the companion present they persist across generators, else they fall back locally.</p>

  <h2 id="debug">Live debug feed</h2>
  <p>Besides <code>start/thought/action/observation/final</code>, <code>onStep</code> also emits <code>\{ type:'prompt', system, transcript, step \}</code> before each model call and <code>\{ type:'raw', reply \}</code> after &mdash; the exact prompt sent and the raw reply received &mdash; so a UI can show a live feed of the model&rsquo;s reasoning. The example page wires these into a debug console.</p>

  <h2 id="fallback">Reasoner &amp; fallback</h2>
  <p>By default reasoning uses the user&rsquo;s own model via <code>sb.ai</code>. With no model configured it returns a clear error; pass <code>opts.reason</code> (a <code>function (prompt, system)</code> returning the model text) to use any reasoner &mdash; a custom model, the broker, or a scripted one for demos. Storage degrades on its own: companion &rarr; <code>weld.persist</code> &rarr; in-memory.</p>
  <div class="note">Design rule, inherited from skybridge: do something reasonable with no companion, something better with one. The agent still runs (in-memory state, a reasoner you supply) when nothing is installed.</div>

  <h2>Changelog</h2>
  <h3>1.1.0</h3>
  <p>Reasoning now requests <strong>native JSON</strong> from your own model, so the one-JSON-object-per-step contract is enforced at the source. Added <code>web.search</code> (companion <code>search</code> capability) so an agent can <em>find</em> pages, not just fetch URLs it already knows. <code>web.fetch</code> extracts readable text from HTML and, given a <code>query</code>, keeps the query-relevant passages instead of head-truncating. <code>memory.search</code> now <strong>ranks</strong> results by relevance instead of returning unordered substring hits. The system prompt nudges a brief plan-first step on multi-step goals and a sanity-check before finalizing.</p>
  <h3>1.0.1</h3>
  <p>The built-in <code>web.fetch</code> tool now falls back to <code>weld.fetch</code> (the superFetch CORS-free proxy) when the companion doesn't provide a <code>fetch</code> capability &mdash; so an agent can browse the web today if the host imports <code>weld-fetch-plugin</code>, with no companion broker required. Response bodies are truncated to 8&nbsp;KB to keep the agent's context manageable.</p>
  <h3>1.0.0</h3>
  <p>Initial release. ReAct loop on weld.skybridge; durable memory, scratchpad, and workspaces; resumable runs; pluggable tool registry with built-ins; robust JSON extraction; live <code>prompt</code>/<code>raw</code> debug events; graceful fallback to a supplied reasoner and <code>weld.persist</code>/memory.</p>
